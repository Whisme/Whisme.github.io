数据库设计（第七章）

7.1.1常用数据类型

  • int：整数，-2^31～2^31-1，精度为10，长度为4字节。

  • bigint：大整数，-2^63～2^63-1，精度为19，长度为8字节。

  • smallint：短整数，-2^15（-32768）～2^15-1（32767），精度5，2字节。

  • tinyint：微短整数，0～255，1字节，精度为3。

  • bit：存储1、0或NULL，1字节，相当于逻辑型数据，TRUE转为1，
    FALSE转为0；bit类型数据赋0时，其值为0，而赋非0（如3），其值为1。

7.1创建完整性约束(6种)

  • 主键(PRIMARY KEY)约束：

    • 在表中定义一个主键值，主键列不允许取空值。

  • 唯一性（UNIQUE）约束：

    • 某列或多列不能存在重复记录行数据，允许取空值。

  • 检查（CHECK）约束：

    • 用来限制用户输入某一列数据范围

  • 默认值（DEFAULT）约束：

    • 在表中插入数据时，若某列没指定数据，该列就
      输入一个默认值。

  • 外键（FOREIGN KEY）约束：

    • 表和表之间的引用关系通过外键约束实现。

  • 空值/非空值（NULL/NOT NULL）约束：

    • 表中某列数据存在不确定，就设置该列为允许空值（NULL），创建表时，默认为允许空值。

列约束和表约束

  • 默认值约束（列约束 ）

    • [CONSTRAINT <约束名>] DEFAULT <常量表达式>。

  • 空值/非空值约束（列约束 ）

    • NULL/NOT NULL

  • 主键约束（列约束，表约束 ）

    • 列约束：[CONSTRAINT <约束名>] PRIMARY KEY

    • [CONSTRAINT <约束名>] PRIMARY KEY (列名1，列名2，…，列名n)

  • 唯一性约束（列约束，表约束 ）

    • 列约束：[CONSTRAINT <约束名>]UNIQUE

    • 表约束： [CONSTRAINT <约束名>] UNIQUE(列名1，列名2，…，列名n)

  • 外键约束

    • 列级约束：[CONSTRAINT <约束名>] [FOREIGN KEY] REFERENCES <主表名>(<主键>)

    • 表级约束：[CONSTRAINT <约束名>] FOREIGN KEY(外键 ) REFERENCES <主表名>(主键)

  • 检查约束

    • 列级约束：[CONSTRAINT <约束名>]CHECK(<只涉及一个列名的逻辑表达式>)

    • 表级约束：[CONSTRAINT <约束名>] CHECK(涉及多个列名的逻辑表达式)

补充内容

  • USE <数据库名>：设置当前操作的数据库.

  • Go语句：批处理的结束标志，单独作为一行

网络边界

  • 网络边界是指内部安全网络和外部非安全网络的分界线。

3.1.1虚拟防火墙技术特点

  • 每个虚拟防火墙独立维护一组安全区域

  • 每个虚拟防火墙独立维护一组资源对象

  • 每个虚拟防火墙独立维护自己的包过滤策略

  • 每个虚拟防火墙独立维护自己的ASPF策略、NAT策略、ALG策略

  • 可限制每个虚拟防火墙占用资源数，如防火墙Session以ASPFSession数目。

3.1.2虚拟服务器

  • 虚拟服务器（ virtual server）是一种模拟物理服务器的虚拟化软件。

  • 基本特性:多实例、隔离性、封装性（考试必考-了解每个特性指的内容P29页）

  • 虚拟服务器优点：

    • 实时迁移
    • 快速部署
    • 高兼容性
    • 提高资源利用率
    • 动态调度资源

3.1.3云存储设备

云存储设备（cloud storage device）机制是指专门为基于云配置所设计的的存储设备。这些设备的实例可以被虚拟化。主要分为四个单位:

  • 文件（file）:数据以文件的形式存放在文件夹中

  • 块（block）:存储的最低等级，是可以被独立访问的最小数据单位

  • 数据集（dataset) :基于表格的以分隔符间隔或以记录形式组织的数据集合对象

  • （object） :将数据及其相关的元数据组织为基于WEB的资源，各种类型的数据都可作为WEB资源被引用和存储。

云存储的安全问题

数据的安全性、完整性、保密性

  • 用户的操作安全

4.4数据库设计-逻辑结构设计

什么是逻辑结构设计？

  • 是将E-R图转换为关系模式的过程

    逻辑结构设计的任务：

  • 逻辑结构设计任务:将E-R图转换成某个具体的数据管理系统所支持的数据模式,并建立相应的外模式。

    逻辑结构设计步骤：

  • (1）将E-R图转化为关系数据模型

  • (2）关系模式的优化

  • (3）设计用户外模式(视图)

4.4.1将E-R图转化为关系数据模型

  • 将E-R图转化为关系数据模型实际上是要将实体、属性和实体之间的联系和主码转化为关系模式。

    转化过程中的原则：

  • 实体的转换

    • 一个实体转化为一个关系模式。实体的属性就是关系的属性。实体的主码就是关系的主码。

  • 联系的转换

    • 可转换为独立的一个关系模式,也可合并到实体的关系模式。

      具体的转换规则

  • 1：1联系 的转换规则

    • 联系的属性其中一方的主键加入另一方实体集对应的关系模式中。

  • 1：N联系的转换规则

    • 把1端关系的主码和联系自身的属性加入到N端关系模式中。

  • M：N联系的转换规则

    • 联系是新的一个关系模式，他的属性是M：N端实体的主码加上联系的属性，其主码是两端实体主码的组合。

  • 两个以上的M:N联系的转换规则

    • 两个以上的实体集合的多元联系可以转换为一个新的关系模式，其属性是各个相连实体的主码，以及联系自身的属性，他的主码是各个实体主码的组合，或指定单独属性作主码。

全局E—R图转换为关系模式的步骤（特别重点）

  • 1、写出1端实体的关系模式;

  • 2、写出与1端连接的1端或者多端实体关系模式;

  • 3、写出多对多联系的实体的关系模式;

  • 4、最后写出多对多联系的关系模式。

数据库设计（第四章）

4.1数据库设计概述

  • 数据库设计

    • 根据用户需求设计数据库结构的过程

数据库设计方法

  • 按照规范设计方法

    数据库设计的六个阶段

  1. 需求分析（重点）
  2. 概念结构设计（重点）
  3. 逻辑结构设计（重点）
  4. 数据库物理设计
  5. 数据库实施
  6. 数据库的运行和维护

4.2需求分析

  • 需求分析阶段的重要性

    1. 掌握用户的需求，理解用户的需求。
    2. 需求分析的结果是否准确地反映了用户的实际要求。
    3. 需求分析的作用要远远大于其它各个阶段

  • 明确需求分析的任务

    1. 详细调查现实世界要处理的对象，明确用户的各种需求，并在此基础上确定新系统的功能。（要根据用户需求设计）
    2. 考虑今后可能的扩充和改变

  • 需求分析的重点是数据和处理

    1. 数据要求：
      • 确定在数据库中需要存储哪些数据。
    2. 处理要求：
      • 指用户要完成什么处理功能。
    3. 安全与完整性要求：
      • 安全性要求是指对数据库的用户、角色、权限、加密方法等安
        全保密措施的要求
      • 完整性要求是指对数据取值范围、数据之间
        各种联系的要求等等。

4.2.2需求分析的方法

  1. 调查组织机构情况
  2. 调查各部门的业务活动情况
  3. 明确用户对系统的各种要求（数据、处理、安全与完整性）
  4. 确定新系统的边界（计算机什么可以做，什么不能做）
  5. 产生数据流图、功能模块图及数据字典等，编写系统需求分析报告。

4.3概念结构设计

  • 主要步骤（三步）

    1. 设计局部E-R 图
    2. 设计全局E-R 图：将局部E-R 图集成为全局概念模式
    3. 评审

局部E-R图设计

  • 确定局部结构范围：

    1. 将整体任务划分成几个系统，又把每个系统划分成几个子系统。
      • 按照功能划分
      • 按照业务逻辑划分
      • 按照部门划分

    • 划分原则：

      1. 独立性原则：
        • 划分在一个范围内的应用功能具有独立性与完整性，与其他范围内的应用有最少的联系。
      2. 规模适度原则：
        • 局部E-R图规模应适度，一般以6个左右实体为宜。

  • 确定实体、属性、主键

    1. 根据问题语义和用户需求识别实体中存在的固有属性
      • 实体：一般需要多个属性进行描述
      • 属性：可以用一个值表达清楚

  • 确定实体间联系及其属性

    • 根据问题语义或者事物规则确定实体间联系
      1. 一对一联系
      2. 一对多联系
      3. 多对多联系
    • 联系本身可以有属性，当一个属性不宜属于任一实体时，可定
      义为实体之间联系的属性。

4.3全局E-R图设计流程

  • 确定公共实体→消除冲突→合并→全局优化

  1. 确定公共实体
    • 同名实体，具有相同键的实体
  2. 消除冲突
    • 局部E-R模型之间不一致的地方，称之为冲突。
    • 冲突分为属性冲突，结构冲突和命名冲突。
      • 属性冲突
        • 属性值的类型、取值范围或取值集合不同。
      • 结构冲突
        • 同一对象在不同应用中的不同抽象。
        • 同一实体在不同局部E-R图中属性组成不同
        • 实体间联系在不同局部E-R图中呈现不同的类型。
    • 命名冲突
      • 包括属性名、实体名、联系名之间的冲突。
        • 同名异义，即不同意义的对象具有相同的名字；
        • 异名同义，即同一意义的对象具有不同的名字。

4.3局部E-R图的合并

  • 合并原则：从公共实体开始，两两合并，先合并那些有联系的局部
    结构，最后再加入独立的局部结构。

3.4.1 完全函数依赖和部分函数依赖

完全函数依赖

  • 例如：选课（学号，姓名，课程号，成绩）

    • （学号，课程号）→成绩

    • 学号唯一对应成绩/课程号也唯一对应成绩

部分依赖函数

  • 例如：选课（学号，姓名，课程号，成绩）

    • （学号，课程号）→姓名

    • 只有学号唯一对应姓名

传递依赖函数

  • 例如：学生（学号，姓名，性别，年龄，所在系，系主任）

    • 通过学号查找到学生所在系，再通过所在系判断系主任

    • 所以学号→系主任是传递函数依赖

非平凡函数依赖

  • 例如：课程（学生号，课程号，成绩）

    • （学生号，课程号）X→成绩Y

    • 成绩依赖于学生号和课程号，但成绩不是他们的子集，所以是非平凡函数依赖

平凡函数依赖

  • 例如：课程（学生号，课程号，成绩）

    • （学生号，课程号）X→课程号Y

    • 课程号依赖于学生号和课程号，但课程号Y是（学生号，课程号）X的子集

非平凡和平凡函数依赖总结：看Y是不是X的子集就行了

  • 是子集——平凡函数依赖

  • 不是子集——非平凡函数依赖

3.4.2关系规范化的目的

  • 避免以下情况出现

    • 数据冗余大

    • 操作异常

    • 更新异常

    • 插入异常

    • 删除异常

  • 一个好的关系模式：无操作异常，数据冗余尽量少

什么是范式

  • 范式(Normal Forms，简记NF)一个规则序列，衡量关系模式的标准。

  • 关系模式的规范化按照从低级到高级的等级分:第一范式(1NF),第二范式_（2NF)，第三范式（3NF）和BCNF，第四范式和第五范式等等。满足1NF的关系称为规范化关系。

什么是关系规范化

  • 低一级范式的关系模式，通过模式分解可以转换为若干个高一级范式的关系模式的集合，这个过程称为关系规范化。

什么是最小函数依赖集FD

  • 概念：

    • (1）FD集中任何一个函数依赖的右部仅含有一个属性。

    • (2）FD集中所有函数依赖的左边都没有冗余属性。

    • (3）FD集中不存在冗余的函数依赖。

  • 例如：学生关系S（学号，姓名,性别,课程号,成绩)关系S中最小函数依赖集为:

    • 最小函数依赖集：

      • 学号→姓名,学号→性别，(学号，课程号）→成绩;

    • 存在冗余的函数依赖：

      • (学号，课程号）→姓名;

第一范式

  • 关系模式R的所有属性都是不可再分的基本数据项

第二范式

  • 每个非主属性完全函数依赖于码

  • 主关键字是属性的组合,这样的关系模式可能不是第二范式。

第三范式

  • 每个非主属性都不传递依赖于R的码

贝叶斯决策

基本概念

  • 1.先验概率
    • 预先已知或可以估计的某种类型的概率
      • 某地癌症发生率为5%
        • 正常类：P=（ω 1）=99.5%
        • 异常类：P=（ω 2）=0.5%
  • 2.类条件概率
    • 某一类x出现的概率，例如这个x可以是阳性或阴性。
      • 正常类出现的概率为1%，异常类出现的概率为95%
        • 正常类：P=（x|ω1）=1%
        • 异常类：P=（x|ω2）=95%
  • 3.后验概率
    • x条件下位于某种类型的概率
      • 例如出现数据x且正常类的概率是67.7%，那么异常类的概率就是32.3%。
        • 正常类：P=（ω1|x）=67.7%
        • 异常类：P=（ω2|x）=32.3%

最小错误率

  • 哪一类的概率大就分到哪一类（判断x属于哪一类的概率最大）

最小风险条件

  • 哪一类风险小就选择哪一类
